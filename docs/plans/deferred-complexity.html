Profiles and downloads

Keep Downloads/Offline and Multi-profile out of the core path until user-state persistence, profile migration, and playback source-origin seams exist.

IndexPrevious: Live TVmulti-profile.mddownloads-offline.mddownloads-offline-spec.mdfavorites.mdvideo-player.md
Deliberately deferred

Decision

Do not add a dormant ProfileStore, DownloadManager, download queue tables, manifests, file-storage abstractions, profile picker, or profile-aware wrappers until the active features consume them. First land provider-scoped favorites/watch/search preferences. Then migrate to one active local profile named Primary. Only then implement narrow Downloads v1.

Dependency order

OrderCapabilityWhy
1Provider-scoped user stateFavorites/watch/search preferences prove content keys and cleanup behavior before profiles widen the key.
2Basic Multi-profile migrationCreate one active local profile, migrate legacy rows to Primary, and add profile ID to persisted user state.
3Playback source-origin seamDownloads need to know whether a player source is remote provider URL, local file, or unsupported collection.
4Downloads v1Only direct VOD/episode files with reliable URLs, file manifest, integrity state, storage checks, and local playback fallback.

Downloads v1 narrow scope

  • Supported: direct movie/episode VOD files only, when the provider exposes a downloadable/playable URL.
  • Unsupported: live streams, DRM-protected content, server-side catch-up, provider requiring token refresh for every segment, cross-device sync.
  • Required state: queue item, transfer status, local file manifest, size/checksum when available, source metadata, provider/profile ownership.
  • Required UX: queued/downloading/paused/failed/completed, retry/remove/delete, storage error copy, play local file when present.

Profile v1 narrow scope

  • Single active local profile after migration, default named Primary.
  • Profile switch updates Favorites, Continue Watching, Search recents, Downloads, and playback preferences.
  • No PIN locks, parental restrictions, cloud sync, or per-profile provider credentials in v1.

Spec cross-links

  • Basic Multi-profile Spec: owns profile migration and active profile behavior.
  • Downloads and Offline and Downloads and Offline Spec: own download UX and storage contracts.
  • Favorites + For You plan: prerequisite provider-scoped user-state rows.
  • Playback plan: prerequisite source/progress seam.
  • Foundation plan: active-provider cleanup semantics that profiles/downloads must respect.

How to handle these workstreams

Treat profile/download UI as explicit unavailable states until the prerequisites above exist. The worst outcome is a half-built manager/table abstraction that no working feature uses and that future profile/download work must unwind.

Reviewer notes

This page intentionally delays ambitious functionality in favor of a clean dependency chain. It protects the current app from a second architecture convention.
