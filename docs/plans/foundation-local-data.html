Foundation and local data

Stabilize provider/session/catalog rules before adding richer UX. The current app is a single-active-provider SQLiteData/GRDB app; the plan keeps that simple and makes the invariants explicit.

Recommended first

Decision

Do not introduce profiles, multi-provider catalogs, a search index, favorites/progress/download rows, EPG, or rich metadata in this workstream. First make the existing one-provider catalog predictable:

  • ProviderManager owns one active Session.
  • SyncManager destructively refreshes the active local catalog on provider edit/switch/delete/initial sync.
  • Category and Media are the active provider catalog, not a historical multi-provider cache.
  • Initial sync stays category-first; stream rows hydrate lazily by selected category.
  • Prefix visibility moves from provider-keyed UserDefaults into a provider-scoped local table.

Why this comes first

Prevents wrong data joins

Current local rows do not carry durable provider/profile ownership. Adding user state before clarifying identity risks orphaned favorites/progress/download records.

Unblocks real UX

Browse, Search, Details, For You, Live, and Downloads all depend on knowing whether a row belongs to the active provider and whether it is hydrated.

Keeps architecture boring

The app already works around SQLiteData/GRDB. This plan avoids reintroducing SwiftData or a parallel cache/service convention.

After this lands, go straight to Library UX. Do not start For You, Live, Favorites, Downloads, profiles, or EPG until Browse/Search can accurately report local catalog coverage and hydration state.
