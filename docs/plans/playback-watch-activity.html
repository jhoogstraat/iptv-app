P0/P1 · Playback reliability

Playback and watch activity

Make playback dependable and persist resumable progress before building richer discovery, favorites, downloads, or profile behavior.

IndexPrevious: Library UXNext: Favorites + For Youvideo-player.mdadvanced-player-ui-spec.mdmedia-details.mdfor-you.md
Core user-state seam

Decision

Add one provider-scoped local watch-activity model and wire real player events to it. Keep playback preferences device-global in UserDefaults until profiles exist. Do not introduce profile-aware preference stores, recommendation indexes, episode switching, or offline playback in this workstream.

Implementation phases

PhaseWorkAcceptance
1Add a provider-scoped watch_activity table/store keyed by stable content identity: media type + remote source ID, with snapshot title/artwork fields.Progress survives app restart and provider-scoped cleanup hides stale rows.
2Wire Player progress and ended events to the store. Replace no-op/commented progress hooks with real writes.Playback updates current time, duration, completed status, and last-watched timestamp without blocking UI.
3Resume eligible movies/episodes from stored progress.Starting playback resumes from unfinished progress and starts from zero for completed or too-short progress.
4Clean advanced player controls so UI only exposes capabilities backed by current backend/data.No visible controls silently do nothing; unsupported controls are disabled or hidden with clear copy.
5Feed watch activity into detail affordances and later For You.Detail pages can show resume state; For You can consume the same rows without a new recommendation service.

Guardrails

  • Series collections are not directly playable; route to episode rows once episodes exist.
  • Keep VLC-first/AV fallback behavior through PlaybackBackend and MediaPlaybackSourceResolver.
  • Do not store profile IDs yet; profiles are deferred and must migrate this table later.
  • Do not implement Downloads/offline fallback here. Offline requires manifest, file integrity, source-origin, and storage state.
  • Do not replace the stable PlayerView shell; swap renderer/backend-specific surfaces underneath it.

How to handle the next workstream

After progress persistence works end to end, implement persisted favorites and For You. Continue Watching should read this table directly; do not create a second recommendation/progress projection.

Reviewer notes

The current player already has real backend and URL-resolution plumbing. Treat this as a state/persistence integration pass, not a player rewrite.
