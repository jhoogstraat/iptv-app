Live TV

Implement the smallest useful Live TV surface: local live categories/channels and playback through the existing player. EPG, catch-up, zapping, DVR, and guide caching wait.

Channel-only first

Decision

Add MediaType.live and live category/channel sync using the same local catalog pattern as Movies/Series. Live channels route through the shared Player and Xtream URL resolver. Do not show fake guide rows or empty EPG shells.

Implementation phases

PhaseWorkAcceptance
1Extend schema/model/mapper/sync for live categories and channel rows.Initial/live sync can persist live categories and lazily hydrate channel rows without breaking movie/series sync.
2Replace LiveScreen placeholder with local category/group/channel browsing and local title search.Live tab displays synced channel categories, filter/search state, and honest empty/loading/failure states.
3Resolve live playback URLs through existing provider/session/player plumbing.Selecting a channel launches playback via shared player with live-specific unsupported controls hidden.
4Update counts, tabs, labels, filters, and docs affected by the new media type.No switch/count/filter silently ignores .live where Live is in scope.

Deferred live features

  • EPG program cache and current/next program display.
  • Catch-up playback from program time windows.
  • Channel zapping and ordered remote-control navigation.
  • DVR/recording/timeshift.
  • Global Live search expansion.

How to handle the next workstream

After channel browsing and live playback work end to end, create a separate Live EPG Foundation plan: provider-scoped program cache, EPG-by-channel/time-window sync, current/next program display, and graceful missing-guide behavior. Catch-up and zapping should follow only after program/time-window data exists.

Reviewer notes

The main risk is coupling guide UI to channel browsing too early. Ship channel-only Live first so the app has a real IPTV surface before guide complexity.
